# ADR-001: Incremental Four-Phase Refactor (Not Big-Bang Rewrite)

## Status

Accepted

## Date

2026-06-21

## Context

The SIMPA frontend needs both **faster initial load** for dashboard users and **lower maintenance cost** for the development team. Analysis identified duplication (read-only catalogs, CRUD patterns), large monolithic files (CSS, enrichment forms, view utils), and a single 882 KB JS chunk because all routes load at startup.

Three product-level delivery strategies were considered. The planning session with the stakeholder selected an incremental roadmap over a full rewrite or performance-only work.

## Decision

Deliver frontend maintainability and performance improvements in **four sequential phases**, each shippable independently with its own success criteria and test gates. Preserve existing user-facing behavior; refactors must not change business rules, API contracts, or dashboard JSON semantics.

Phase order: (1) quick wins — catalogs, lazy routes, CSS; (2) CRUD standardization and chunk tuning; (3) split large modules; (4) registry, tests, documentation.

## Alternatives Considered

### Alternative 1: Big-bang frontend rewrite

- **Description**: Rebuild Cadastros/Admin/Importação on a new feature-folder architecture in one release.
- **Pros**: Clean slate; uniform patterns from day one.
- **Cons**: High regression risk; blocks other features for weeks; E2E and production validation cost.
- **Why rejected**: Importação and Cadastros are production-critical with Playwright coverage; stakeholder requires continuous delivery.

### Alternative 2: Performance-only (code splitting, no structural refactor)

- **Description**: Add `React.lazy` and Vite `manualChunks` only; leave duplicated pages and god files untouched.
- **Pros**: Fastest path to smaller bundle; minimal code churn.
- **Cons**: Duplication and file size problems remain; next catalog page still copy-paste; PR review burden unchanged.
- **Why rejected**: Stakeholder explicitly asked for maintainability analysis, not only bundle warning fix.

### Alternative 3: Incremental four-phase refactor (chosen)

- **Description**: Phased deduplication, splitting, and lazy loading with per-phase metrics.
- **Pros**: Balances user benefit (load time) and team velocity; each phase verifiable; aligns with existing Compozy task workflow.
- **Cons**: Longer calendar time than performance-only; requires discipline to avoid scope creep per phase.
- **Why accepted**: Matches approved planning roadmap and SIMPA's history of Compozy-delivered features.

## Consequences

### Positive

- Predictable delivery milestones aligned with Compozy tasks.
- Reduced risk compared to rewrite; E2E can run after each phase.
- Developers gain reusable patterns (`ReadOnlyCatalogPage`, `useEntityCrud`) for future SIA/cadastro work.

### Negative

- Temporary coexistence of old and new patterns until phase 2–3 complete.
- Multiple PRs/branches to manage.

### Risks

- Silent UI regressions during refactor — mitigate with Vitest, existing Playwright specs, and manual smoke per module.
- Over-abstraction — mitigate with YAGNI; do not merge Importação into generic CRUD.

## Implementation Notes

- Detailed HOW belongs in `_techspec.md` and task files.
- Phase 1 should land before phase 3 file splits to avoid merge conflicts on the same hotspots.
