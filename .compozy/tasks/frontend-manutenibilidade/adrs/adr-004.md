# ADR-004: useEntityCrud Hook (Not Extending CadastroCrudPage)

## Status

Accepted

## Date

2026-06-21

## Context

Three CRUD flavors exist: config-driven `CadastroCrudPage` (equipes, emendas), bespoke `UsuariosPage`, bespoke `IndicadoresPainelPage` (with preview modal). Shared state: list load, form open/edit, confirm inactivate, busy id, toast.

## Decision

Extract **`hooks/useEntityCrud.ts`** with injected API callbacks and optional extensions (e.g., preview slot for widgets). Refactor `UsuariosPage` and `IndicadoresPainelPage` to use it. **Do not** force `CadastroCrudPage` to absorb admin/painel-specific UX.

`CadastroCrudPage` may optionally adopt the hook internally in a follow-up if duplication remains.

## Alternatives Considered

### Extend CadastroCrudPage with plugins

- **Pros:** Single page component.
- **Cons:** Widget preview, discovery, metric picker do not fit entity config schema.
- **Why rejected:** Hook + page composition is more flexible.

### Duplicate state machines

- **Pros:** No abstraction risk.
- **Cons:** Perpetuates 175–388 line pages.
- **Why rejected:** PRD F4.

## Consequences

### Positive

- Bug fixes in dialog flow apply to Admin and Painel widgets cadastro.

### Negative

- Hook API must support optional delete vs inactivate naming (`inactivateItem` vs `deleteItem`).
