# ADR-001: Dual-Table Hybrid Storage + Full Hospital Panel MVP

## Status

Accepted

## Date

2026-06-24

## Context

SIHD (Sistema de Informações Hospitalares — Decisor) exposes two MySQL tables in the `producao` database:

- **`s_aih`** — one row per hospitalization (AIH header): dates, diagnoses, length of stay, UTI days, total value, discharge reason, complexity, specialty.
- **`s_aih_pa`** — multiple rows per AIH (procedure items): procedure code, CBO, quantity, unit value, financing detail.

The system needs to import this data into PostgreSQL for the Hospital panel. Three storage strategies were evaluated.

## Decision

**Hybrid dual-table storage:**

1. `sih_internacoes` — stores `s_aih` data aggregated at the managerial grain: `competencia × CNES × PROC_PRINCIPAL × DIAG_PRINCIPAL × COMPLEXIDADE × FINANCIAMENTO × MOTIVO_SAIDA × SEXO_PACIENTE`. Metrics summed: `qtd_aih`, `total_diarias`, `total_diarias_uti`, `total_valor`. Average: `media_idade`, `media_diarias`.

2. `sih_procedimentos` — stores `s_aih_pa` items aggregated at `competencia × CNES × PROC_DETALHADO × CBO_PROFISSIONAL × FINANCIAMENTO_DETALHE`. Metrics: `qtd_aih_distintas`, `total_quantidade`, `total_valor_item`.

The Hospital panel MVP (Layout A) exposes all KPI categories: gerencial (AIH + diárias + valor), epidemiológico (CID chapters, mortalidade, motivo saída), and procedimentos (s_aih_pa ranking).

## Alternatives Considered

### Alternative B: Full-Fidelity 1 Row = 1 AIH

- **Description**: Copy every `s_aih` row as-is into PostgreSQL; aggregate only `s_aih_pa`.
- **Pros**: Patient-level granularity; can calculate true length of stay per AIH.
- **Cons**: Volume proportional to raw AIH count per municipality; complex PG queries for panel aggregations.
- **Why rejected**: Managerial panel only needs aggregated views; `DIARIAS` and `VALOR_TOTAL_AIH` are pre-calculated by SIHD, so row-level precision adds no benefit for the target use case.

### Alternative C: MVP Minimal — s_aih Aggregated Only

- **Description**: Import only `s_aih` aggregated in Phase 1; defer `s_aih_pa` and CID/financing KPIs to Phase 2.
- **Pros**: Fewer tables, faster implementation.
- **Cons**: Hospital panel incomplete at launch; requires a second sync round for procedures.
- **Why rejected**: User confirmed full KPI set is required in MVP.

## Consequences

### Positive

- Consistent with SIA pattern (`sia_producao` = aggregated s_prd).
- PG volume minimized via GROUP BY aggregation in MySQL before insert.
- All Hospital panel KPIs available in a single import cycle.
- Independent tables allow KPIs from `sih_internacoes` to render without waiting for `sih_procedimentos` subset.

### Negative

- Two tables and two extraction queries per sync cycle (one for s_aih, one for s_aih_pa).
- Re-sync of a competência must DELETE + reinsert both tables (handled by reimport gate).

### Risks

- JOIN `s_aih ↔ s_aih_pa` uses three columns (AIH + CNES + COMPETENCIA) with COLLATE utf8mb4_unicode_ci — must be enforced in extraction query.
- `FINANCIAMENTO` in `s_aih` is 2-char (maps directly to `RUB_ID`), unlike SIA's 4-char `PRD_RUB` — collation and key length differ.
