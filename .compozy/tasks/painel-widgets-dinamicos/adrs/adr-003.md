# ADR-003: Server-Side Named Placeholder Binding for SQL Templates

## Status

Accepted

## Date

2026-06-20

## Context

Seed metrics store SQL with named placeholders (`:competencia`, `:estabelecimento_id`, `:equipe_id`). Node `pg` driver uses positional `$1`, `$2`. Templates must never concatenate user input into SQL strings.

## Decision

Implement **`bindTemplate(sql, params)`** in `painelMetricsService.js`:

1. Load `sql_template` only from `painel_metricas_catalogo` by ID (never from request body).
2. Replace allowed named tokens with positional `$n` in fixed order: `competencia`, `estabelecimento_id`, `equipe_id`.
3. Pass values as typed parameters (`competencia` → first day of month `DATE`; IDs → `bigint` or `null`).
4. Reject templates containing tokens outside the whitelist or multiple statements (`;` after first statement).

Optional `fonte_config.fallback_chave` triggers a second catalog lookup if primary query returns no row.

## Alternatives Considered

### Alternative 1: Store positional SQL in seed only

- **Pros**: No binding layer.
- **Cons**: Harder for admins to read `sql_preview`; error-prone migrations.
- **Why rejected**: PRD requires readable SQL preview with named filters.

### Alternative 2: Use a SQL parser library

- **Pros**: Robust validation.
- **Cons**: New dependency; overkill for three placeholders.
- **Why rejected**: YAGNI for MVP.

## Consequences

### Positive

- Templates remain human-readable in cadastro UI.
- Injection surface limited to three bound parameters.

### Negative

- Custom template authoring in future phases must follow naming convention.

### Risks

- **Malformed seed SQL**: Mitigate with integration tests executing all seed templates against fixture DB.
