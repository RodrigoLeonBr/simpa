# ADR-002: Dedicated Painel Layout Runtime Endpoint

## Status

Accepted

## Date

2026-06-20

## Context

The Painel currently loads a single monolithic JSON document via `GET /api/v1/dashboard/planejamento`. Dynamic widgets require loading configuration from `painel_widgets`, resolving multiple metric templates against PostgreSQL, and returning a structure tailored for `KpiCard` and ECharts renderers.

Options: extend the existing dashboard response, embed widget resolution inside `fetchDashboard`, or expose a separate runtime endpoint.

## Decision

Add **`GET /api/v1/dashboard/painel-layout`** that returns resolved widgets for `(perfil, layout, competencia, estabelecimento_id?, equipe_id?)`.

Keep **`/planejamento` unchanged** for SIA/SIHD module status, legacy contract fields, and municipal aggregation paths used elsewhere (`/metas`, `/indicadores`).

`LayoutA` will call both endpoints in parallel when `painelPerfil === 'APS'` and layout is `A`; module status bar continues to use `/planejamento` data.

## Alternatives Considered

### Alternative 1: Extend `/planejamento` JSON with `widgets[]`

- **Pros**: Single round-trip; simpler frontend hook.
- **Cons**: Bloats contract v3.1.0; couples ETL JSON envelope to SQL widget engine; increases payload for pages that do not need widgets.
- **Why rejected**: Violates separation between consolidated contract and configurable layout layer.

### Alternative 2: Frontend resolves metrics

- **Pros**: One config fetch, client-side cache of catalog.
- **Cons**: Exposes SQL templates to browser; cannot enforce server-side governance; duplicates executor logic.
- **Why rejected**: Security and governance requirements mandate server execution only.

## Consequences

### Positive

- Clear boundary: configurators edit via `/api/cadastros/painel-*`; consumers read via `/v1/dashboard/painel-layout`.
- `/planejamento` consumers unaffected.
- Easier to add caching/batching on layout endpoint later.

### Negative

- Painel APS Layout A performs two HTTP requests (acceptable; parallel fetch).
- Frontend must merge loading/error states from both sources.

### Risks

- **Partial failure**: Mitigate with LayoutA fallback to hardcoded `buildPainelKpis` when layout endpoint fails but `/planejamento` succeeds.

## Implementation Notes

See `_techspec.md` — `usePainelLayout` hook composes layout response with existing `useDashboard` data for module bar.
