# ADR-003: Denominator Integration via pop_row Parameter in build_payload()

## Status

Accepted

## Date

2026-06-22

## Context

`etl_contract.py::build_payload()` currently returns hardcoded `"—"` for `num` and `den` in all 13 indicators. Population denominators must be injected for C1, B4, IGM-APS, IGM-PN, IGM-VAC, IGM-ICSAP. Three integration points were considered: passing a `pop_row` dict as a parameter, querying the DB inside `_build_indicadores_qualidade()`, or running a separate post-consolidation UPDATE pass.

## Decision

Add an optional `pop_row: dict | None = None` parameter to `build_payload()`. `consolidate_group()` in `consolidate_dashboard.py` queries `populacao_cadastrada` for the matching `(competencia, estabelecimento_id)` before calling `build_payload()` and passes the result. `_build_indicadores_qualidade(pop_row)` computes denominators from `pop_row` fields when available, falling back to `"—"` when `pop_row` is `None`.

## Alternatives Considered

### Alternative: Query inside _build_indicadores_qualidade()

- **Description**: Pass `conn` (or `estabelecimento_id` + `competencia`) to `_build_indicadores_qualidade()` and query `populacao_cadastrada` there.
- **Pros**: Self-contained; no caller changes needed.
- **Cons**: `etl_contract.py` becomes coupled to the database. Currently it is a pure data-transformation module with no I/O. Breaking this invariant makes unit testing harder and the module less reusable.
- **Why rejected**: Keeping `etl_contract.py` I/O-free is a design invariant worth preserving.

### Alternative: Post-consolidation UPDATE pass

- **Description**: Run `build_payload()` without population data → write to `dados_consolidados` → then UPDATE `dados_consolidados.dados_conteudo` JSONB to inject `den` values.
- **Pros**: No changes to `build_payload()` signature.
- **Cons**: Two database writes per consolidation group; `dados_consolidados` temporarily holds incomplete data; race condition if API reads between write and UPDATE.
- **Why rejected**: Two-phase write is complex and introduces transient inconsistency.

## Consequences

### Positive

- `etl_contract.py` remains a pure transformation module (no DB access).
- Denominators are computed in a single consolidation pass.
- `pop_row=None` fallback means existing consolidation works unchanged for groups without cadastro individual data.
- Unit tests for `_build_indicadores_qualidade()` can pass a mock `pop_row` dict.

### Negative

- `build_payload()` signature grows by one optional parameter.
- `consolidate_group()` gains an extra DB query per group (cheap: keyed lookup by `(competencia, estabelecimento_id)`).
