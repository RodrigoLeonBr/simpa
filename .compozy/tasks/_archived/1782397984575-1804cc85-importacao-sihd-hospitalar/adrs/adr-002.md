# ADR-002: Standalone sync_sih_mysql.py ETL Script

## Status

Accepted

## Date

2026-06-24

## Context

SIHD data lives in two MySQL tables (`s_aih`, `s_aih_pa`) that have a different schema, different collation requirements, and different aggregation grain compared to SIA (`s_prd`). The existing `sync_sia_mysql.py` contains helpers shared by both systems (connection management via `etl_db.py`, `execute_batch` wrappers, progress event format, SAVEPOINT error isolation). A decision is needed on whether to keep one script or two.

## Decision

Create a standalone `sync_sih_mysql.py` that mirrors the structure of `sync_sia_mysql.py` but targets `s_aih`/`s_aih_pa`. Both scripts share `etl_db.py` for connection management and use the same progress event JSON format for Node.js streaming. No `--modulo` flag or conditional branching inside a single script.

## Alternatives Considered

### Alternative: Extend sync_sia_mysql.py with --modulo sia|sih

- **Description**: Single script, conditionally switches query, table names, and grão based on `--modulo` flag.
- **Pros**: Shared helpers in one file; fewer files to maintain.
- **Cons**: SIA and SIH have fundamentally different schemas (varchar with CAST vs int/decimal; 2-char vs 4-char financiamento; two source tables vs one); branching logic would make both paths harder to read and test independently. Risk of SIA regression when touching SIHD code.
- **Why rejected**: Coupling SIA and SIH in one script violates the single-responsibility principle and increases risk of SIA regression.

## Consequences

### Positive

- SIA sync remains unchanged; no regression risk.
- SIHD ETL script is fully readable without mental context-switching.
- Independent test suites for SIA and SIH.
- Follows project convention established by `parse_esus_csv.py` vs `sync_sia_mysql.py` (separate scripts per source system).

### Negative

- Duplication of connection boilerplate (mitigated by `etl_db.py` shared module).
- Two scripts to update if the progress event JSON schema changes.

### Risks

- `etl_db.py` interface must remain stable; any breaking change requires updating both scripts.

## Implementation Notes

`sync_sih_mysql.py` CLI interface mirrors `sync_sia_mysql.py`:
- `--competencia YYYY-MM` (required for single-month sync)
- `--pg-write` (actual insert; omit for dry-run JSON preview)
- `--reimportar` (DELETE + reinsert)
- `--exec-id UUID` (telemetry)
- `--json-out` (print result JSON to stdout)

Environment variable prefix: `SIH_*` (e.g., `SIH_TABLE_AIH`, `SIH_PG_BATCH_SIZE`).
