# ADR-003: SIHD Import UI in /importacao Page

## Status

Accepted

## Date

2026-06-24

## Context

The SIHD import operation needs a UI entry point. Two existing patterns exist in SIMPA:
1. **Cadastros page** — hosts `CadastroSyncBanner` (reference data sync) and `SiaProducaoSyncBanner` (SIA production sync). Banner-style inline sections.
2. **Importação page (`/importacao`)** — hosts the e-SUS file upload workflow, with history and status tracking.

SIHD import is a MySQL sync operation (not a file upload), but it involves a year/month selector, progress polling, and sync history — closer in UX complexity to the SIA production banner than to e-SUS upload.

## Decision

Place the SIHD import UI as a new section (`SihImportSection.tsx`) within the existing `/importacao` page, grouped alongside other import operations. The Cadastros page will show a read-only SIHD sync status badge linking to `/importacao` for the full operation.

## Alternatives Considered

### Alternative A: Banner in Cadastros (below SiaProducaoSyncBanner)

- **Description**: `SihImportBanner.tsx` in the Cadastros page, same pattern as `SiaProducaoSyncBanner`.
- **Pros**: Consistent with SIA sync location; less navigation for planning staff.
- **Cons**: Cadastros page already has multiple sync banners (cadastros reference + SIA production). Adding a third makes the page crowded. SIHD is a different data domain (hospital vs outpatient).
- **Why rejected**: User confirmed preference for Importação page; cleaner separation of concerns.

### Alternative B: Dedicated page /importacao/sihd

- **Description**: Standalone page for SIHD import with full history and status.
- **Pros**: Maximum isolation; clean URL.
- **Cons**: Extra route and navigation item; adds complexity for a section that shares structure with existing importação flows.
- **Why rejected**: Overkill for MVP; the /importacao page already supports multiple import types.

## Consequences

### Positive

- `/importacao` becomes the single home for all import operations (e-SUS, SIA, SIHD).
- No new route entry needed in navigation config.
- Cadastros page stays focused on reference data management.

### Negative

- Planning staff who currently manage everything in Cadastros must switch to `/importacao` for SIHD sync.
- Requires modest refactor of the Importação page to accommodate a new section.

### Risks

- If `/importacao` page grows further (future SINAN, SIM), layout may need restructuring.

## Implementation Notes

- `SihImportSection.tsx` lives at `simpa-frontend/src/pages/Importacao/SihImportSection.tsx`.
- Cadastros page adds a `SihSyncStatusBadge` (read-only, links to `/importacao`) so planning staff can see SIHD sync status without leaving Cadastros.
- `/importacao` page renders `<SihImportSection />` below the e-SUS section with a visual divider.
