# ADR-002: ReadOnlyCatalogPage + usePaginatedCatalog Hook

## Status

Accepted

## Date

2026-06-21

## Context

FormasPage, CbosPage, and ProcedimentosPage duplicate search state, pagination, loading/error handling, and table layout. `buildFormasQuery`, `buildCbosQuery`, and `buildProcedimentosQuery` in `enrichmentView.ts` are identical.

## Decision

Introduce:

- `buildPaginatedCatalogQuery(q, page, extra?)` in `utils/enrichmentView.ts` (or `utils/catalogView.ts` if enrichmentView grows).
- `hooks/usePaginatedCatalog.ts` — generic paginated fetch state.
- `components/cadastros/ReadOnlyCatalogPage.tsx` — config-driven UI shell.

Entity pages become thin wrappers exporting column defs + fetch function + copy.

## Alternatives Considered

### Keep duplicated pages

- **Pros:** Zero refactor risk.
- **Cons:** Every new SIA catalog copies ~120 lines.
- **Why rejected:** PRD goal F1.

### Extend CadastroCrudPage for read-only mode

- **Pros:** One CRUD component family.
- **Cons:** CadastroCrudPage is edit-focused; read-only has no FormDialog; wrong abstraction.
- **Why rejected:** Separate read-only catalog clearer.

## Consequences

### Positive

- ~250 LOC removed; EstabelecimentosPage can reuse hook later with extra filters.

### Negative

- Props interface must stay stable for future catalogs (formas, cbos, procedimentos, future SIH lists).
