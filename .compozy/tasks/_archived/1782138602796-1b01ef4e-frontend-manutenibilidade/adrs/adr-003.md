# ADR-003: Route-Level Code Splitting with React.lazy

## Status

Accepted

## Date

2026-06-21

## Context

`App.tsx` statically imports all page modules. Production bundle is 882 KB (277 KB gzip). ECharts loads even for users who only open Painel.

Industry practice (2025–2026): route-based splitting first; lazy heavy chart libs second; avoid over-splitting small components.

## Decision

1. **Phase 1:** `React.lazy` + `Suspense` in `App.tsx` for:
   - `CadastrosPage` (`/cadastros/*`)
   - `ImportacaoPage` (`/importacao`)
   - `AdminPage` (`/admin/*`)
2. Keep **Login**, **Painel**, **AppShell** eager.
3. **Phase 2:** Lazy-load `MetasPage`, `IndicadoresPage`, `RelatoriosPage`; add Vite `manualChunks` for `react`/`react-router-dom` and `echarts/*`; dynamic import `EChart.tsx` where used.
4. Add lightweight fallback UI (`ModuleLoadingFallback` component).
5. Wrap lazy routes in existing or new `ErrorBoundary` for chunk load failures (retry message in PT).

## Alternatives Considered

### Component-level splitting only

- **Pros:** Finer control.
- **Cons:** Smaller gzip win; more boundaries to maintain.
- **Why rejected:** Route split gives largest initial bundle reduction.

### No splitting — only refactor files

- **Pros:** Simplest code changes.
- **Cons:** Does not meet PRD load-time goal.
- **Why rejected:** ADR-001 requires both maintainability and performance.

## Consequences

### Positive

- Painel-first users download less JS; aligns with Vite code-splitting defaults at dynamic `import()`.

### Negative

- Brief loading state on first navigation to Cadastros/Admin; must not flash unstyled content.

### Risks

- Stale chunk after deploy — mitigate with error boundary + hard refresh hint.
