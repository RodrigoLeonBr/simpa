# ADR-003: SIA MySQL via XAMPP Host

## Status

Accepted

## Date

2026-06-19

## Context

SIA/SUS production data lives in a MySQL instance on the municipal XAMPP server. The PRD requires read-only access to tables `s_prd`, `prestador`, and `procedimento`. Docker Compose runs the SIMPA API in a container; MySQL remains on the host.

## Decision

Connect the **api** container to host MySQL using:

- **Windows**: `host.docker.internal` as `MYSQL_HOST`
- **Linux**: `extra_hosts: ["host.docker.internal:host-gateway"]` in compose
- **Credentials**: read-only MySQL user in `.env` (`MYSQL_USER`, `MYSQL_PASS`, `MYSQL_DB`)
- **Script**: `sync_sia_mysql.py --competencia YYYY-MM --pg-write` invoked by API `POST /api/sia/sincronizar`

Sync writes to PostgreSQL tables `sia_sincronizacoes` and `sia_producao`; consolidation merges SIA into `dados_consolidados`.

## Alternatives Considered

### Alternative 1: MySQL in Docker Compose

- **Description**: Dedicated MySQL container with periodic dump import
- **Pros**: Pure container networking
- **Cons**: Stale data vs live XAMPP; duplicate maintenance
- **Why rejected**: User chose host XAMPP as source

### Alternative 2: Defer SIA to Phase 2

- **Description**: MVP without SIA connector
- **Pros**: Faster initial delivery
- **Cons**: MAC tab and ambulatorial module remain empty
- **Why rejected**: Full redesign scope includes SIA sync

## Consequences

### Positive

- Reads live SIA data without migration
- Connector isolated in `sync_sia_mysql.py`; API only spawns subprocess

### Negative

- Container-to-host networking is environment-specific
- XAMPP instability (PRD risk) can break sync

### Risks

- **Risk**: MySQL not listening on accessible interface — **Mitigation**: bind-address docs for XAMPP
- **Risk**: Schema differences in municipal SIA — **Mitigation**: configurable table/column mapping in script

## Implementation Notes

- API health check optionally probes MySQL connectivity
- Dashboard returns `status_conexao: "MySQL_XAMPP_CONNECTED" | "MySQL_XAMPP_UNAVAILABLE"`
