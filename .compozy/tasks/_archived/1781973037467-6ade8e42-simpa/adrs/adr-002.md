# ADR-002: Spec-Driven Stack (Node + React + Python + PostgreSQL)

## Status

Accepted

## Date

2026-06-19

## Context

The PRD defines a spec-driven architecture: JSON contract v3.1.0 is the single source of truth between ETL, API, and frontend. Existing artifacts include `parse_esus_csv.py`, `schema_full.sql`, and plans A/B/C. No Laravel or Go code exists in the repository.

## Decision

Adopt the PRD-recommended stack without substitution:

| Layer | Technology |
|-------|------------|
| Database | PostgreSQL 15+ with JSONB (`dados_consolidados.dados_conteudo`) |
| ETL | Python 3.11 — `parse_esus_csv.py`, `consolidate_dashboard.py`, `sync_sia_mysql.py` |
| API | Node.js 18 + Express 4 (CommonJS) |
| Frontend | React 18 + Vite + TypeScript + Tailwind + ECharts |
| Auth | JWT (`jsonwebtoken`) + bcrypt |

The API orchestrates Python via subprocess (stdout JSON), never reimplements parsing logic in Node.

## Alternatives Considered

### Alternative 1: Laravel monolith (PHP in XAMPP)

- **Description**: Single PHP app with Blade/React Inertia
- **Pros**: Native XAMPP integration
- **Cons**: Diverges from all existing specs, ETL, and schema; no PHP codebase exists
- **Why rejected**: Would discard months of PRD and Python ETL work

### Alternative 2: NestJS instead of Express

- **Description**: TypeScript backend with decorators and DI
- **Pros**: Strong typing, modular structure
- **Cons**: Heavier boilerplate for a small team; Plano B already specifies Express
- **Why rejected**: YAGNI — Express matches existing plans and skill level

## Consequences

### Positive

- Frontend can start with json-server mock using contract v3.1.0
- Python ETL remains independently testable
- JSONB extensibility preserved for future modules

### Negative

- Three runtimes to maintain (Node, Python, PostgreSQL)
- Subprocess ETL adds latency on upload (acceptable for monthly batch)

### Risks

- **Risk**: Contract drift between ETL and API — **Mitigation**: JSON schema validation in tests; `versao_schema` field

## Implementation Notes

- Shared `.env` for PG and MySQL credentials
- `null` semantics: never coerce to zero in API or frontend
