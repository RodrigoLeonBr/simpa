# ADR-001: Curated Metric Catalog with Governed SQL Templates

## Status

Accepted

## Date

2026-06-20

## Context

SIMPA's APS Painel (Layout A) displays six KPI cards and two charts whose definitions are hardcoded in frontend utilities. Planning staff need to choose which indicators appear on the Painel and which production values they reflect, without waiting for developer releases.

Brainstorm evaluated three data-binding strategies:
- Reading pre-consolidated JSON fields only (limited to ETL output).
- Querying live database metrics through a curated catalog (user chose this).
- Allowing free-form SQL from the admin UI (rejected on governance and safety grounds).

The municipal health BI market (Painel e-SUS APS, Planeja+APS, commercial e-SUS dashboards) expects territory-level views (município, UBS, equipe), regulated quality indicators, and governed metric definitions — not ad hoc report building by every user.

Schema Phase 0 already created `painel_metricas_catalogo` and `painel_widgets` with seed data mirroring the current Layout A.

## Decision

Adopt a **two-layer model**:

1. **Metric catalog** — a flat, discoverable list of approved metrics (e-SUS raw rows, SIA production, consolidated JSON paths). Each entry carries a human label, source metadata, and a **parameterized SQL template** bound to Painel filters (competência, estabelecimento, equipe).

2. **Widget configuration** — planning staff assemble Painel slots (card, line chart, ranking) by picking metrics from the catalog, setting titles and display format, and optionally viewing the SQL template for transparency. Configuration is restricted to Administrador and Planejamento roles.

Metric discovery from imported e-SUS loads populates the catalog (manual trigger in MVP; optional automation after import in a later phase). SQL templates are not user-editable in MVP — preview is read-only.

MVP delivers **both** the cadastro experience and a dynamic APS Layout A Painel in the same release (equal priority).

## Alternatives Considered

### Alternative 1: JSON contract paths only

- **Description**: Widgets resolve values from `dados_consolidados` JSON without live SQL.
- **Pros**: Simpler, aligns with existing `/planejamento` API, no extra queries.
- **Cons**: Cannot expose raw e-SUS metrics not yet in the consolidator; planning staff cannot tie widgets directly to import lineage; contradicts stakeholder choice for live database values.
- **Why rejected**: Does not meet the stated requirement for database-backed, configurable indicators.

### Alternative 2: Free-form SQL editor for admins

- **Description**: Power users write arbitrary SELECT statements per widget.
- **Pros**: Maximum flexibility; satisfies "see the SQL" literally.
- **Cons**: High governance risk; inconsistent definitions; difficult audit; performance and security exposure in a municipal health context.
- **Why rejected**: SQL visibility is required for **transparency**, not unrestricted editing. Templates provide preview without opening injection vectors.

### Alternative 3: Cadastro-first, Painel read-only seed later

- **Description**: Ship configuration UI before dynamic Painel rendering.
- **Pros**: Smaller initial UI scope; backend can be validated in isolation.
- **Cons**: Planning staff configure widgets with no visible outcome; low adoption feedback loop; splits user value across releases.
- **Why rejected**: Stakeholder selected equal priority for cadastro autonomy **and** live Painel parity in MVP Phase 1.

## Consequences

### Positive

- Planning staff gain self-service control over APS Layout A composition within governed bounds.
- Metric definitions become reusable across widgets and auditable via catalog entries.
- Parity with current Painel is preserved via seed data while enabling future customization.
- Aligns with industry pattern: curated catalog + role-based configuration (not self-serve raw SQL).

### Negative

- Each Painel load may execute multiple queries (performance consideration for later optimization).
- Municipal vs unit-level filter semantics require careful template design (some current seeds assume municipal aggregation).
- Two parallel indicator concepts exist today (`indicadores` table from migration 003 vs `painel_metricas_catalogo`) — integration deferred.

### Risks

- **Misconfiguration**: Mitigate with preview-before-save, role restriction, and seed defaults.
- **Catalog drift after new e-SUS report shapes**: Mitigate with discovery job and occurrence counters.
- **User confusion between `/indicadores` analytics page and Painel widgets**: Mitigate with clear naming in cadastro ("Indicadores do Painel") and UX copy.
