# ADR-003: Profile-Specific Enrichment in Four Physical Tables

## Status

Accepted

## Date

2026-06-20

## Context

Today `estabelecimentos.enriquecimento JSONB` stores hospital fields (`leitos`, `especialidades`, `habilitacoes`, `notas`) and the API allows enrichment only for Hospitalar/Misto. PRD requires distinct enrichment schemas per profile (APS, MAC, Hospitalar, Misto). Stakeholder chose normalized storage over nested JSONB keys.

## Decision

Introduce four 1:1 tables keyed by `estabelecimento_id`:

| Table | Columns (MVP minimum) |
|-------|----------------------|
| `enriquecimento_aps` | `notas_territorio`, `cobertura_populacional`, `vinculo_esus`, `prioridades_planejamento`, `notas` |
| `enriquecimento_mac` | `capacidades` (TEXT[]), `relacionamento_referencia`, `autorizacoes`, `notas` |
| `enriquecimento_hospitalar` | `leitos` (JSONB), `especialidades` (TEXT[]), `habilitacoes` (TEXT[]), `capacidade_notas`, `notas` |
| `enriquecimento_misto` | `leitos` (JSONB, optional), `capacidades_ambulatoriais`, `notas_mac`, `notas` |

**Outro** profile: `enriquecimento_outro` with `notas TEXT` only (fifth small table; same access pattern).

- Deprecate writes to `estabelecimentos.enriquecimento` JSONB; keep column read-only during migration window.
- One-time migration copies existing JSONB hospital data → `enriquecimento_hospitalar` where `perfil IN ('Hospitalar','Misto')`.
- API: `PUT /estabelecimentos/:id/enriquecimento/:perfilSlug` where slug ∈ `aps|mac|hospitalar|misto|outro`.
- GET establishment joins active enrichment row for current `perfil` only (or returns all for admin drawer).

## Alternatives Considered

### Alternative 1: Nested JSONB keys per profile in existing column

- **Pros:** No migration; matches archived ADR-003.
- **Cons:** Weak validation; harder to query/report.
- **Why rejected:** Stakeholder chose normalized tables.

### Alternative 2: Single `estabelecimento_enriquecimento` with `perfil` discriminator

- **Pros:** One table, simpler migrations.
- **Cons:** Sparse columns or generic JSONB payload negates normalization benefit.
- **Why rejected:** Stakeholder chose four physical tables.

## Consequences

### Positive

- Strong per-profile validation in service layer.
- Hospital bed schema unchanged in spirit (`leitos` JSONB on hospitalar table).
- Clear extension point for Phase 2 reporting.

### Negative

- Migration + five new tables; more JOINs on GET detail.
- Legacy `PUT .../enriquecimento` must redirect or alias to hospitalar slug.

### Risks

- **Risk:** Orphan enrichment rows after perfil change — **Mitigation:** on perfil PUT, do not auto-delete old enrichment; drawer loads schema for new perfil (old row retained for audit/revert).

## Implementation Notes

- `estabelecimentosService.js` gains `getEnrichmentForPerfil`, `upsertEnrichmentForPerfil`.
- Remove `ENRICHMENT_PERFIS` gate; validate body shape per table.
- Frontend: replace single `EnrichmentForm` with `EnrichmentFormByPerfil` switch.
