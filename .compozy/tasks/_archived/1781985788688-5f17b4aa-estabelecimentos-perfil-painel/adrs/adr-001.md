# ADR-001: Editable Establishment Profile with Phased Multi-Profile Dashboard

## Status

Accepted

## Date

2026-06-20

## Context

SIMPA already mirrors MySQL `prestador` into `estabelecimentos` and derives `perfil` from `tipouni` at sync time. The field is locked in the UI even though it is not a native MySQL column — it is a SIMPA classification. Establishments can be filtered by profile in Cadastros, but enrichment is limited to Hospitalar/Misto and the Painel loads only APS units while offering layout switchers A/B/C without a profile dimension.

Stakeholders want `perfil` to be user-editable, profile-aware enrichment in Cadastros, and the Painel to source unit lists from the establishment registry with a profile selector (APS, MAC, Hospitalar, Misto) driving distinct indicator sets across all three layouts. MVP must deliver full APS dashboard value before MAC, Hospitalar, and Misto indicator packs ship.

## Decision

Adopt a **unified profile model** spanning Cadastros and Painel:

1. **Cadastros (Phase 1 — full scope in MVP)**
   - Make `perfil` editable by Administrador, Gestor Secretaria, and Planejamento.
   - On MySQL re-sync, **preserve user-edited `perfil`**; sync updates identity fields only.
   - Provide **profile-specific enrichment** for APS, MAC, Hospitalar, and Misto (not hospital-only).
   - Keep profile filter chips on the establishments list; drawer shows editable perfil + enrichment form matched to selected perfil.

2. **Painel (Phase 1 — APS indicators complete; other profiles scaffolded)**
   - Add a **profile selector** (APS, MAC, Hospitalar, Misto) alongside existing layout switcher A/B/C.
   - Populate unit filters and ranking/table rows from **active establishments** matching the selected profile.
   - Each profile × layout combination uses a **distinct KPI catalog** (not shared labels with different values).
   - **MVP ships complete indicator packs only for APS** across layouts A, B, and C.
   - MAC, Hospitalar, and Misto show profile selector and unit lists but display placeholder or reduced views until Phase 2 indicator definitions are approved.

## Alternatives Considered

### Alternative 1: Cadastro-first, Painel deferred (B)

- **Description**: Deliver editable perfil and per-profile enrichment now; postpone all Painel multi-profile work.
- **Pros**: Lower delivery risk; faster Cadastros win.
- **Cons**: Painel remains APS-only; planning staff cannot act on the new profile taxonomy in their primary workspace.
- **Why rejected**: User needs Painel profile selector in the same initiative; phased indicator delivery balances risk.

### Alternative 2: Big-bang four-profile dashboard (C)

- **Description**: Ship Cadastros + full distinct KPI sets for APS, MAC, Hospitalar, and Misto simultaneously.
- **Pros**: Coherent launch across all care levels.
- **Cons**: Requires four indicator catalogs before any release; high definition and validation burden; delays value.
- **Why rejected**: Stakeholder chose APS-complete MVP with other profiles following.

### Alternative 3: Overwrite perfil on sync

- **Description**: Keep derived perfil from MySQL as source of truth.
- **Pros**: No drift between SIA type and SIMPA classification.
- **Cons**: Negates user intent when municipal mapping is wrong; blocks the stated need to edit perfil.
- **Why rejected**: User explicitly chose preserve manual edits on sync.

## Consequences

### Positive

- Single establishment registry becomes the source for both Cadastros filtering and Painel unit scope.
- Planning staff can correct misclassified establishments without fighting sync.
- Profile-specific enrichment supports differentiated planning context per care level.
- Phased Painel rollout delivers APS value while setting UX pattern for MAC/Hospitalar/Misto.

### Negative

- `perfil` may diverge from `tipouni`-derived defaults; governance relies on role-restricted editing.
- Four enrichment schemas increase UX and validation surface.
- Non-APS Painel profiles may feel incomplete until Phase 2 — requires clear empty/coming-soon states.

### Risks

- **Risk**: Users expect full MAC/Hospitalar dashboards at launch — **Mitigation**: visible profile tabs with explicit "indicators em definição" for non-APS until Phase 2.
- **Risk**: Inconsistent enrichment quality across profiles — **Mitigation**: define minimum field sets per profile in PRD; iterate with Planning Unit workshops in Phase 2.
- **Risk**: Misto establishments double-counted across profiles — **Mitigation**: Misto is its own Painel profile; establishments appear only under Misto when so classified.

## Implementation Notes

- Builds on archived `simpa-cadastros-sync` ADR-003 (`estabelecimentos` + JSONB enrichment); extends business rules rather than replacing table design.
- Prior design replaced Painel profile tabs with layout A/B/C; this ADR reintroduces profile as an orthogonal axis to layout.
- Indicator catalogs per profile are product-owned definitions; TechSpec will map them to existing dashboard contract fields.
