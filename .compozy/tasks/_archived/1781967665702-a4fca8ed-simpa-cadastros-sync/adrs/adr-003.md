# ADR-003: Single `estabelecimentos` Table with JSONB Enrichment

## Status

Accepted

## Date

2026-06-20

## Context

PostgreSQL currently stores establishments across three manual CRUD tables: `unidades_saude`, `prestadores_mac`, and `hospitais`. The PRD requires one unified establishment entity keyed by MySQL `prestador.re_cunid`, with hospital-specific fields editable only in SIMPA. `equipes` and `metas_financiamento` reference `unidades_saude(id)`.

## Decision

Introduce **`estabelecimentos`** as the canonical establishment table:

| Column group | Storage |
|--------------|---------|
| MySQL identity (locked) | Scalar columns: `codigo_externo`, `nome`, `cnpj`, `re_tipo`, `tipouni`, `perfil`, `area`, `relatorio`, `status`, `sincronizado_em` |
| SIMPA enrichment (editable) | `enriquecimento JSONB` — keys: `leitos` (object by type), `especialidades` (string[]), `habilitacoes` (string[]), `notas` (string) |

- `codigo_externo` UNIQUE NOT NULL maps to `re_cunid`.
- `perfil` derived at sync time via configurable mapping (`CADASTRO_PERFIL_MAP` env JSON).
- Migrate `equipes.unidade_id` → `equipes.estabelecimento_id` (FK to `estabelecimentos`).
- Migrate `metas_financiamento.unidade_id` similarly.
- Drop CRUD routes for `unidades`, `prestadores-mac`, `hospitais`; expose `estabelecimentos` + enrichment PUT only.
- Deprecate old tables after one-time data migration in `migration_004_cadastros_sync.sql`.

## Alternatives Considered

### Alternative 1: Extend `unidades_saude` in place

- **Pros**: Minimal FK churn.
- **Cons**: Table name misleading; MAC/hospital rows never fit semantically; merges three legacy shapes awkwardly.
- **Why rejected**: PRD unified Establishments model needs neutral naming.

### Alternative 2: Normalized enrichment tables (`estabelecimento_leitos`, etc.)

- **Pros**: Relational queries on bed counts.
- **Cons**: Over-engineered for MVP; hospital enrichment is sparse and display-only initially.
- **Why rejected**: YAGNI; JSONB sufficient for Phase 1 dashboards.

## Consequences

### Positive

- Single FK target for equipes/metas.
- Re-sync updates scalar MySQL fields without touching `enriquecimento` (UPDATE excludes JSONB column).
- Profile filter is a simple `WHERE perfil = $1`.

### Negative

- Requires migration script for existing seed/manual rows.
- JSONB enrichment lacks DB-level validation (mitigated by API schema validation).

### Risks

- **Risk**: Orphan equipes after migration if codigo mismatch — **Mitigation**: migration report + manual review queue in sync JSON output.

## Implementation Notes

- `cnes` displayed as alias of `codigo_externo` in UI until separate CNES column is required (PRD open question).
- Inactive MySQL rows (`ativo=0`) → `status='inativo'`; default API list filters `status='ativo'`.
