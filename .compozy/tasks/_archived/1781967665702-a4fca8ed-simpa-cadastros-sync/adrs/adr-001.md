# ADR-001: Unified Establishment Mirror from MySQL Prestador with Enrichment

## Status

Accepted

## Date

2026-06-20

## Context

The municipal SIMPA Cadastros module currently exposes six independent CRUD screens (Units, Teams, Procedures, MAC Providers, Hospitals, Parliamentary Amendments). Meanwhile, the production MySQL database on XAMPP already maintains authoritative master data in `prestador` (establishments/providers) and `procedimento` (SIGTAP catalog). The SIA production sync already reads these tables but does not populate SIMPA cadastros, causing duplicate data entry and drift between billing systems and planning dashboards.

Stakeholders require a single source of truth: aggregate what already exists in MySQL, allow enrichment only where SIMPA adds planning value, and keep SIMPA-native records (teams, amendments, indicator/metas catalog) as manual cadastros.

## Decision

Adopt a **Unified Mirror + Enrichment** product model:

1. **Establishments (Estabelecimentos)** — one cadastro derived from MySQL `prestador`, with profile derived from existing type fields (`re_tipo`, `tipouni`) covering APS, MAC, and Hospitalar contexts previously split across three cards.
2. **Procedures (Procedimentos)** — read-only mirror of MySQL `procedimento`; no manual creation of SIGTAP codes in SIMPA.
3. **MySQL-sourced fields are locked** after sync; users may edit **SIMPA extension fields** only (e.g., hospital bed counts, specialties, authorizations/habilitações, internal notes/links).
4. **Manual sync trigger** — users explicitly run “Update cadastros from SIA” when they want refreshed master data (no automatic schedule in MVP).
5. **SIMPA-native cadastros remain manual** — Teams (e-SUS), Parliamentary Amendments, and Indicators/Metas catalog.

Retire separate UI cards for Units, MAC Providers, and Hospitals in favor of the unified Establishments experience.

## Alternatives Considered

### Alternative 1: Phased — Procedures + Units first, unification later

- **Description**: Sync only two entities in MVP; defer MAC/Hospital merge.
- **Pros**: Smaller initial delivery.
- **Cons**: Users continue seeing duplicate cards and inconsistent mental model during MVP.
- **Why rejected**: Stakeholder chose full unification in one product increment.

### Alternative 2: Separate cards fed by same MySQL source

- **Description**: Keep three cards but auto-fill from `prestador`.
- **Pros**: Minimal UI change.
- **Cons**: Same establishment may appear in multiple lists; still feels like duplicate cadastros.
- **Why rejected**: Stakeholder chose single unified Establishments cadastro.

### Alternative 3: CNES national TXT import as primary enrichment source

- **Description**: Supplement local MySQL with monthly CNES files for beds and authorizations.
- **Pros**: Richer hospital metadata without manual typing.
- **Cons**: Additional import workflow and file management; not required for MVP value.
- **Why rejected**: Deferred; manual enrichment on top of MySQL mirror is sufficient for Phase 1.

## Consequences

### Positive

- Eliminates duplicate entry for establishment names, CNES codes, and SIGTAP procedures already maintained in production MySQL.
- Aligns planning filters and dashboards with the same identifiers used in SIA billing (`re_cunid`, SIGTAP codes).
- Clear separation: synced truth vs. SIMPA planning extensions vs. fully local cadastros.

### Negative

- Users lose ability to create arbitrary establishments/procedures in SIMPA that do not exist in MySQL (by design).
- Profile classification (APS vs MAC vs Hospitalar) depends on mapping municipal `prestador` type codes — requires validation with domain experts.
- Hospital enrichment fields start empty until users fill them or a future CNES import is added.

### Risks

- **Risk**: MySQL schema or type-code semantics differ from assumptions — **Mitigation**: document mapping with municipal SIA operator before build; configurable mapping in TechSpec.
- **Risk**: Inactive establishments (`ativo=0`) clutter UI — **Mitigation**: default view shows active only; optional filter for inactive.
- **Risk**: e-SUS import unit names may not match `prestador` names — **Mitigation**: Open Question for reconciliation UX in PRD.

## Implementation Notes

- This ADR defines product behavior only. Sync mechanics, storage shape, and API contracts belong in TechSpec.
- Existing SIMPA Task 16 UI must be redesigned; current six-card CRUD is superseded for synced entities.
- `sync_sia_mysql.py` already joins `prestador` and `procedimento`; cadastro sync should reuse the same read-only MySQL connection pattern (ADR-003).
