# ADR-002: Dedicated Python Script for Cadastro Sync

## Status

Accepted

## Date

2026-06-20

## Context

MySQL master data for establishments (`prestador`) and procedures (`procedimento`) must be mirrored into PostgreSQL on user-triggered sync. The existing `sync_sia_mysql.py` reads the same MySQL tables but writes production aggregates to `sia_producao` scoped by competencia. Cadastro sync is competencia-independent, upserts catalog rows, and must not overwrite SIMPA enrichment fields.

## Decision

Add a **separate script** `sync_cadastros_mysql.py` that:

- Reuses `etl_db.mysql_connect()` / `pg_connect()` and SIA column env overrides (`SIA_TABLE_PRESTADOR`, etc.).
- Reads full `prestador` and `procedimento` snapshots (no competencia filter).
- Upserts PostgreSQL `estabelecimentos` and `procedimentos` by business keys (`codigo_externo`, `codigo_sigtap`).
- Returns JSON summary stdout for API subprocess parsing (same contract as SIA sync).

Express service `cadastrosSync.js` mirrors `sia.js` spawn pattern.

## Alternatives Considered

### Alternative 1: Extend `sync_sia_mysql.py` with `--cadastros` flag

- **Pros**: One script, shared MySQL connection setup.
- **Cons**: Mixed responsibilities; production sync would accidentally trigger catalog refresh; harder to test independently.
- **Why rejected**: Violates single-responsibility; cadastro sync is on-demand and competencia-agnostic.

### Alternative 2: Sync cadastros inside Node.js via `mysql2`

- **Pros**: No subprocess.
- **Cons**: Breaks established ETL boundary (Node orchestrates, Python reads/writes bulk data); duplicates connection config.
- **Why rejected**: Inconsistent with ADR-002 spec-driven stack.

## Consequences

### Positive

- Clear separation from SIA production pipeline.
- Independent unit tests in pytest.
- API can expose dedicated `POST /api/cadastros/sincronizar` without competencia param.

### Negative

- Another script to maintain alongside `sync_sia_mysql.py`.
- Shared env var naming may confuse operators (mitigated by documentation).

### Risks

- **Risk**: Drift between prestador column mapping in two scripts — **Mitigation**: extract shared mapping constants module `mysql_prestador_map.py` if duplication exceeds 30 lines.

## Implementation Notes

- CLI: `python sync_cadastros_mysql.py --pg-write` (stdout JSON on success).
- Optional `--dry-run` for preview counts without PG write (testing).
