# ADR-002: Dedicated GET /api/populacao Endpoint

## Status

Accepted

## Date

2026-06-22

## Context

The population dashboard at `/painel/populacao` needs to fetch `populacao_cadastrada` data for display (pyramid, conditions, summary cards). Three options exist: a new dedicated endpoint, extending the existing `/api/v1/dashboard/planejamento` payload, or embedding inside the current `indicadores_qualidade` contract structure.

## Decision

Implement a new `GET /api/populacao` route in `simpa-backend/src/routes/` that reads directly from `populacao_cadastrada` joined with `esus_cargas` and `estabelecimentos`. Accepts query params `competencia` (YYYY-MM) and `estabelecimento_id`. Returns structured JSON with `cidadaos_ativos`, `saidas`, `faixa_etaria`, `condicoes_saude`, `raca_cor`, and `sociodemografico`.

## Alternatives Considered

### Alternative: Extend /api/v1/dashboard/planejamento

- **Description**: Add `populacao_cadastrada` key to the ContratoDashboard JSONB payload built by `etl_contract.py`.
- **Pros**: Single API call for Painel; population data co-located with indicators.
- **Cons**: The dashboard payload is per-`(competencia, unidade, equipe)` group; population data is per-`(competencia, estabelecimento_id)` only. Forcing population into the contract bloats `dados_consolidados` and couples the population lifecycle to the consolidation pipeline â€” a failed consolidation would block population data from being served.
- **Why rejected**: Population data is a snapshot (uploaded once per competĂŞncia per unit) and should be queryable independently of whether production reports have been consolidated.

### Alternative: Include in indicadores_qualidade contract

- **Description**: Return computed `den` values alongside existing indicator fields; population data orphaned with no standalone endpoint.
- **Pros**: Minimal new code.
- **Cons**: No way to display population panel without fetching the full dashboard payload; cannot support the `/painel/populacao` route independently.
- **Why rejected**: The population panel is a standalone section that must load without depending on dashboard consolidation state.

## Consequences

### Positive

- `/painel/populacao` loads independently of dashboard consolidation.
- Simple query: `SELECT ... FROM populacao_cadastrada WHERE competencia = $1 AND estabelecimento_id = $2`.
- Aggregate across units: query without `estabelecimento_id` filter, sum in service layer.
- Access control: same `verifyJWT` middleware, no new auth logic.

### Negative

- One additional backend route file (or module in existing `api.js` router tree).
- Frontend makes two API calls when displaying Painel with population tab (dashboard + population).
