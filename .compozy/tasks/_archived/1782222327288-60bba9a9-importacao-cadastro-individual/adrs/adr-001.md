# ADR-001: Dedicated Population Table for Cadastro Individual Data

## Status

Accepted

## Date

2026-06-22

## Context

SIMPA needs to store aggregated population data from e-SUS "Relatório de Cadastro Individual Analítico" CSVs. This data serves two purposes: (1) denominators for 13 quality indicators (C1, B1-B6, M1, M2, IGM-APS, IGM-PN, IGM-VAC, IGM-ICSAP) and (2) a population dashboard section in the main Painel. The existing generic EAV table `esus_indicadores_raw` stores raw CSV sections as JSONB but was designed for flow-based reports (atendimentos, procedimentos), not population snapshots. The population dashboard requires structured queries for pyramid charts, condition prevalence, and denominator lookups that are expensive over a generic EAV schema.

## Decision

Store cadastro individual data in a dedicated structured table (`populacao_cadastrada`) with typed columns for active citizens, age/gender breakdown (JSONB array), health conditions (JSONB), and sociodemographic summary (JSONB). The import pipeline extends the existing `esus_cargas` flow (add `cadastro_individual` to CHECK constraint) and a new ETL parser populates the dedicated table. Indicator denominators are derived by querying `populacao_cadastrada` directly.

## Alternatives Considered

### Alternative A: Extend existing pipeline (esus_indicadores_raw)

- **Description**: Add `cadastro_individual` to `esus_cargas.tipo_relatorio` CHECK; store all CSV sections as rows in `esus_indicadores_raw`; derive denominators via JSONB queries in `etl_contract.py`.
- **Pros**: Zero new tables; reuses existing parser logic; same EAV pattern.
- **Cons**: Denominator extraction requires complex multi-row JSONB queries per indicator; population pyramid and condition charts need aggregation over generic EAV, making dashboard queries slow and brittle.
- **Why rejected**: The dashboard population panel needs structured data for visualization. EAV queries for "sum of masculino for faixas 25–64" would be fragile and hard to maintain across 13 indicators.

### Alternative C: Hybrid raw + materialized summary

- **Description**: Store all sections in `esus_indicadores_raw` (raw audit trail) AND materialize key aggregates into a `populacao_resumo` table.
- **Pros**: Raw data auditable; fast denominator lookups.
- **Cons**: Two tables to keep in sync; extra ETL step; raw EAV is redundant if the dedicated table is the source of truth.
- **Why rejected**: Overengineering for MVP. The dedicated table (`populacao_cadastrada`) itself serves as the auditable record since it is sourced directly from the CSV import.

## Consequences

### Positive

- Denominator queries are simple typed lookups (`SELECT condicoes_saude->>'hipertensao' FROM populacao_cadastrada WHERE ...`).
- Population pyramid chart reads directly from `faixa_etaria` JSONB array without aggregation over EAV rows.
- Adding new denominator fields in the future is a schema migration, not a query rewrite.
- Clean FK to `esus_cargas` preserves the import audit trail.

### Negative

- One new table and migration required.
- ETL needs a dedicated parser for cadastro individual (different CSV structure from flow reports).

### Risks

- e-SUS may change section names or order in future PEC versions; parser must be tolerant of missing sections. **Mitigation**: parse by section header string match, store unknown sections as `extras` JSONB field.

## Implementation Notes

- `populacao_cadastrada` should have a UNIQUE constraint on `(carga_id)` since one carga maps to one population snapshot.
- `faixa_etaria` stores array of `{faixa, masculino, feminino}` objects matching the CSV rows exactly.
- `condicoes_saude` stores all Sim/Não/Não informado counts for each condition (hipertensao, diabetes, gestante, fumante, etc.).
- `sociodemografico` stores race/color, disability, education, employment breakdowns.
