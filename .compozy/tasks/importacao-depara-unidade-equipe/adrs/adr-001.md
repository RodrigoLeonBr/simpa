# ADR-001: Mapping Registry with Preview Gate for e-SUS Import

## Status

Accepted

## Date

2026-06-20

## Context

SIMPA imports e-SUS analytic CSV reports where the unit name in the file header (e.g., `CAFI CENTRO DE ASSISTENCIA A FAMILIA E AO IDOSO`) does not match the registered establishment name in the MySQL mirror (e.g., `7169698` / `CAFI - CENTRO DE ASSISTÊNCIA A FAMÍ`, profile APS). Today, `esus_cargas` stores the raw e-SUS string with no link to `estabelecimentos`, so the Panel and consolidation cannot align imported production with registered units and teams.

Stakeholders require a persistent, editable mapping (de-para) reused on future imports, manual establishment selection (no auto-creation of units), automatic team registration when missing, and explicit business rules for the `"Todas"` team filter—including conflict detection when granular team imports replace aggregate `"Todas"` imports within the same month.

## Decision

Adopt a **Mapping Registry + Preview Gate** product approach:

1. Persist editable mappings from e-SUS labels to SIMPA cadastro records (establishment and, when applicable, team).
2. During import preview, apply saved mappings automatically; for unknown labels, suggest candidates by name similarity and require manual confirmation before upload is enabled.
3. Block the **Process** action until every unit and team in the batch is resolved.
4. Auto-create missing teams (not establishments) under the mapped establishment when the CSV carries a specific INE/name.
5. Enforce `"Todas"` rules: allow aggregate import only when no specific-team data exists for that unit in the month; create a team named `"Todas"` when used; on subsequent specific-team import, warn the user and remove the prior `"Todas"` import for that unit/month.

Only planning-staff roles (Administrador, Gestor Secretaria, Planejamento) may create or edit persisted mappings.

## Alternatives Considered

### Alternative 1: Session-only mapping (no central registry)

- **Description**: User maps units per upload session; mappings are not stored for reuse.
- **Pros**: Faster to ship; minimal new UI.
- **Cons**: Repeated manual work every month; high error risk; does not meet stakeholder requirement for reusable de-para.
- **Why rejected**: Explicit requirement to save and edit mappings for future imports.

### Alternative 2: Separate admin screen before any import

- **Description**: Dedicated configuration page where users pre-register all e-SUS aliases before uploading CSVs.
- **Pros**: Centralized maintenance; clear ownership of mappings.
- **Cons**: Users often discover new e-SUS label variants only at export time; forces two-step workflow and delays first import.
- **Why rejected**: Preview-time discovery with persisted registry better matches real municipal workflow.

### Alternative 3: Mapping Registry + Preview Gate (selected)

- **Description**: As described in Decision.
- **Pros**: Reuses mappings; blocks orphan data; supports suggestion + confirmation; fits import UX.
- **Cons**: More UI and state in preview step; requires conflict rules for `"Todas"`.
- **Why selected**: Aligns with all clarified stakeholder answers.

## Consequences

### Positive

- Imported production can be tied to Panel filters and registered establishments.
- Operators map once and reuse aliases across monthly imports.
- `"Todas"` vs specific-team conflicts are handled explicitly, reducing silent data duplication.

### Negative

- First import for a new e-SUS label requires manual mapping even when suggestion is strong.
- `"Todas"` replacement may delete previously consolidated dashboard rows for that unit/month—users must understand the warning.

### Risks

- **Wrong mapping accepted**: Mitigated by name suggestions plus mandatory confirmation on first use; editable registry for correction.
- **Establishment not in cadastro**: User cannot proceed until MySQL sync includes the unit—communicate clearly in preview.

## Implementation Notes

- Business rules only in this ADR; technical design belongs in TechSpec.
- Example mapping: e-SUS `CAFI CENTRO DE ASSISTENCIA A FAMILIA E AO IDOSO` → establishment `7169698` (APS).
- Teams created from import should remain editable in Cadastros like manually created teams.
