# ADR-003: Node Orchestration Layer for Import Mapping and ID Propagation

## Status

Accepted

## Date

2026-06-20

## Context

The PRD requires a persistent de-para registry, preview gate, `"Todas"` conflict handling, and writing `estabelecimento_id` / `equipe_id` through `esus_cargas`, `parse_esus_csv.py`, and `consolidate_dashboard.py`. SIMPA already orchestrates ETL via Node `child_process` spawn (`parser.js`, `consolidator.js`) with thin Express routes.

Alternatives: implement mapping entirely in Python; resolve IDs only in SQL views; or split mapping into a separate microservice.

## Decision

Implement mapping, suggestions, team auto-create, and `"Todas"` rules in a **new Node service** `importMappingService.js`, called from `routes/importacao.js`. Extend existing Python CLIs with optional `--estabelecimento-id` and `--equipe-id` arguments passed by Node after resolution—do not duplicate CSV parsing in Node.

Persist mappings in a new PostgreSQL table `esus_import_mapeamentos` via **migration_006**. Add FK columns to `esus_cargas` and `dados_consolidados` in the same migration.

Upload flow: multipart `files[]` + JSON field `resolucoes` (per-file mapping confirmations and `"Todas"` purge consent).

## Alternatives Considered

### Alternative 1: Python-only mapping module

- **Pros:** Single language in ETL path.
- **Cons:** Preview/upload routes, cadastro CRUD, and JWT auth live in Node; would require new Python HTTP layer or heavy IPC.
- **Why rejected:** Breaks existing thin-route → service → spawn pattern.

### Alternative 2: SQL VIEW join at dashboard query time

- **Pros:** No schema change on `esus_cargas`.
- **Cons:** Rejected in PRD ADR-002; consolidation uniqueness still text-based.
- **Why rejected:** Does not meet FK storage requirement.

### Alternative 3: Node orchestration + Python ID flags (selected)

- **Pros:** Minimal change surface; reuses parser/consolidator; mapping near auth and cadastro services.
- **Cons:** Cross-process contract (CLI args) must stay in sync.
- **Why selected:** Matches codebase conventions and PRD.

## Consequences

### Positive

- One service owns de-para CRUD, preview enrichment, upload gate, and `"Todas"` purge.
- Python scripts remain focused on CSV I/O and SQL writes.

### Negative

- `parser.js` / `consolidator.js` signatures grow; integration tests must cover spawn args.

### Risks

- **Drift between Node resolution and Python write:** Mitigate with shared integration test importing CAFI fixture end-to-end.
- **Partial upload failure in batch:** Use per-file transactions in Node orchestration; document rollback behavior in TechSpec.

## Implementation Notes

- Name similarity: Node utility normalizing ASCII (reuse logic aligned with `parse_esus_csv.normalize_key` semantics).
- Synthetic `"Todas"` team codigo: `TODAS-{estabelecimento_id}` to satisfy global `equipes.codigo` UNIQUE.
