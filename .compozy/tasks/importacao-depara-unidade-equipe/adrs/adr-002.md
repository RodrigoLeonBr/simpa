# ADR-002: Cadastro Keys on Import and Consolidation Storage

## Status

Accepted

## Date

2026-06-20

## Context

SIMPA stores e-SUS imports in `esus_cargas` using free-text `unidade` and `equipe_nome` parsed from CSV headers. Consolidated Panel data in `dados_consolidados` uses the same text dimensions with a unique key `(competencia, unidade, equipe)`.

The Panel filter bar loads establishments from cadastro (`estabelecimentos.id`, `nome`) and passes **resolved names** to the dashboard API. When the e-SUS label differs from the registered name—as with CAFI (`CAFI CENTRO DE ASSISTENCIA A FAMILIA E AO IDOSO` vs `CAFI - CENTRO DE ASSISTÊNCIA A FAMÍ`)—the query returns no row and charts stay empty despite successful import.

The upcoming Metas module already models targets with `estabelecimento_id` and `equipe_id` (`metas_financiamento`). Production data without the same keys cannot be joined to targets.

Schema comments in `schema_full.sql` anticipated replacing free text with cadastro references (`unidades_saude` note on `esus_cargas.unidade`) but this was never implemented.

## Decision

Extend the import and consolidation pipeline to **persist cadastro primary keys** alongside retained e-SUS text:

1. **`esus_cargas`:** add `estabelecimento_id` (required after successful mapping) and `equipe_id` (required when team scope applies, including synthetic `"Todas"` team).
2. **`dados_consolidados`:** add `estabelecimento_id` and `equipe_id`; treat these as the authoritative dimensions for Panel filtering and Metas joins. Keep text `unidade` / `equipe` for display and audit.
3. **De-para resolution** (ADR-001) runs before write; keys are set at import time, not inferred later at query time.
4. **Panel consumption:** filter and load dashboard by establishment/team IDs; names are secondary display fields.

Original e-SUS strings remain stored and unchanged for traceability and de-para matching keys.

## Alternatives Considered

### Alternative 1: Name normalization only

- **Description:** Normalize strings (trim, uppercase, alias table) and keep querying by text.
- **Pros:** Smaller schema change.
- **Cons:** Fragile; encoding differences; cannot support Metas FK model; repeated edge cases per unit.
- **Why rejected:** Does not satisfy Panel reliability or Metas requirement.

### Alternative 2: Resolve IDs only at query time (JOIN by de-para)

- **Description:** Keep storage as text; dashboard service JOINs de-para registry when filtering.
- **Pros:** No migration of existing columns.
- **Cons:** Every query depends on mapping table; consolidation uniqueness still text-based; duplicate logic in import, consolidate, panel, metas.
- **Why rejected:** Keys must live where data is stored for integrity and performance.

### Alternative 3: Cadastro keys on storage (selected)

- **Description:** As in Decision.
- **Pros:** Aligns Panel, import, consolidation, Metas; single source of truth per carga.
- **Cons:** Requires migration and backfill for historical rows (Phase 2).
- **Why selected:** Stakeholder requirement; matches existing Metas schema direction.

## Consequences

### Positive

- Panel unit selection by cadastro ID reliably loads imported data.
- Metas phase can share identifiers without a second mapping layer.
- Audit trail preserved via original e-SUS text columns.

### Negative

- Schema migration and updates to parser, consolidator, and dashboard query paths.
- Historical imports need backfill or remain ID-null until mapped.

### Risks

- **NULL keys on legacy rows:** Mitigate with Phase 2 backfill wizard and block new imports without IDs (MVP).
- **UNIQUE constraint drift:** Revisit uniqueness on `(competencia, estabelecimento_id, equipe_id)` vs text tuple during TechSpec.

## Implementation Notes

- TechSpec owns column definitions, migration number, and API parameter changes.
- Example: e-SUS unit `CAFI CENTRO DE ASSISTENCIA A FAMILIA E AO IDOSO` → `estabelecimento_id` pointing to codigo_externo `7169698`.
- Consolidation trigger after import must pass resolved IDs, not raw CSV names alone.
