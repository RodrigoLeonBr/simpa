# ADR-004: Painel Profile Axis in Global Filters + Client-Side KPI Catalogs

## Status

Accepted

## Date

2026-06-20

## Context

Painel currently hardcodes `fetchEstabelecimentosAps()` and layout switcher A/B/C without a profile dimension. Dashboard API returns a single APS-centric `ContratoDashboard` v3.1.0. PRD requires profile selector (APS, MAC, Hospitalar, Misto) orthogonal to layout, with distinct KPI catalogs per profile; MVP ships full APS indicators only.

## Decision

1. **State:** Add `painelPerfil: PainelPerfil` to `useFilters` context (`'APS' | 'MAC' | 'Hospitalar' | 'Misto'`, default `'APS'`). Changing profile resets `unidadeId` and `equipeId`.

2. **Unit lists:** `FilterBar` and `useDashboard` call `fetchEstabelecimentos({ perfil: painelPerfil, limit: 200 })` instead of APS-only helper.

3. **KPI catalogs:** Implement `dashboardView.ts` registry:
   - `PAINEL_KPI_CATALOGS[perfil][layout]` â†’ builder functions.
   - MVP: only `APS` entries call existing `buildPainelKpis`, `buildRanking`, `buildUnitTable`.
   - `MAC`, `Hospitalar`, `Misto`: return `catalogStatus: 'pending'`; Painel renders `PainelProfilePlaceholder` component (not APS KPIs).

4. **Dashboard API:** No backend change in MVP Phase 1. Phase 2 adds optional `?perfil=` when consolidator produces profile-specific payloads.

5. **UI:** New `ProfileSwitcher` component adjacent to `LayoutSwitcher` in `Painel/index.tsx`.

## Alternatives Considered

### Alternative 1: Local state in Painel only

- **Pros:** Smaller blast radius.
- **Cons:** FilterBar unit dropdown out of sync when navigating away and back.
- **Why rejected:** Stakeholder chose global `useFilters`.

### Alternative 2: URL query `?perfil=`

- **Pros:** Shareable links.
- **Cons:** Extra routing work; not requested for MVP.
- **Why rejected:** Defer to Phase 3.

### Alternative 3: Backend-driven KPI catalog endpoint

- **Pros:** Single source of truth.
- **Cons:** Over-engineering for MVP; catalogs are presentation mapping over existing contract.
- **Why rejected:** Client-side registry sufficient for Phase 1 APS.

## Consequences

### Positive

- FilterBar and Painel share one profile source.
- Clear seam for Phase 2 indicator packs.
- No consolidator changes block MVP cadastro work.

### Negative

- Non-APS profiles show placeholder until Phase 2 data exists.
- `dashboardView.ts` grows; mitigate with per-profile files in Phase 2.
